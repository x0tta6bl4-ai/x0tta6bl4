# Архитектурный гайд: Интеграция SPIFFE/SPIRE в Zero-Trust Mesh (2026)

Этот документ описывает практический опыт внедрения идентичности рабочих нагрузок (Workload Identity) в распределенные системы на примере фреймворка **x0tta6bl4**.

## 1. Зачем это нужно?
В классических сетях доступ определяется IP-адресом или статичным API-ключом. В Zero-Trust Mesh:
- **IP не является идентификатором.**
- **Ключи живут не более 24 часов.**
- **Идентичность подтверждается криптографически.**

## 2. Стек технологий
- **SPIRE (SPIFFE Runtime Environment):** Центр выдачи аттестатов.
- **SVID (SPIFFE Verifiable Identity Document):** JWT или X.509 документ, подтверждающий, кто вы.
- **Workload API:** Интерфейс, через который агент получает ключи без секретов в конфигах.

## 3. Этапы внедрения (Чек-лист)
1. **Аттестация узла (Node Attestation):** Проверка "железа" через TPM/TEE или облачные метаданные (AWS/GCP).
2. **Аттестация рабочей нагрузки (Workload Attestation):** Проверка процесса (SHA256 бинаря, UID пользователя).
3. **MaaS Integration:** Передача JWT-SVID в заголовке `X-SPIFFE-JWT-SVID` для авторизации на Control Plane.

## 4. Пример из x0tta6bl4 (MAPE-K Loop)
Наша петля управления автоматически проверяет свежесть SVID:
```python
if is_node_measured_attestation_fresh(node):
    # Допуск разрешен
    pass
else:
    # Запуск экстренной ротации
    rotate_node_runtime_credential(...)
```

## 5. Вывод
Использование SPIRE позволяет полностью исключить "секреты" из переменных окружения и конфиг-файлов, снижая риск компрометации сети до нуля при краже бинарных данных.
