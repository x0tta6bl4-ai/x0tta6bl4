## 🛑 1. Non-Claims & Anti-Scope (What This Framework Does NOT Prove)

To maintain scientific and engineering integrity, execution of this framework **DOES NOT** constitute proof of:
- Absolute cryptographic or physical security;
- Complete anonymity or un-traceability;
- Universal censorship resistance across un-tested DPI vendors;
- Production readiness for un-tested network topologies ($N > 100$ nodes);
- Exemption from underlying operating system kernel bugs.

This framework answers **ONLY** the specific, empirical questions measured by its reproducible test suites under stated conditions.

## 📏 1.1 Statistical Rigor Requirements

### Sample Size Requirements

| Metric | Minimum N | Preferred N | Rationale |
| :--- | :--- | :--- | :--- |
| Latency (RTT, TTFB) | 30 | 100+ | Central Limit Theorem applies at N ≥ 30 |
| Percentiles (p95, p99) | 100 | 500+ | Requires large sample for stable estimates |
| Throughput | 30 | 100+ | Network variability requires repeated measurement |
| Recovery Time | 30 | 50+ | Each run is independent experiment |

**RULE:** Never output p95/p99 when N < 30. Use median and IQR instead.

### Control Groups (Mandatory)

Every benchmark MUST include comparison baselines:

| Group | Description | Purpose |
| :--- | :--- | :--- |
| **Baseline: Direct ISP** | No tunnel, direct connection | Establishes raw network characteristics |
| **Baseline: Basic Proxy** | Standard SOCKS5/HTTP proxy | Shows basic proxy overhead |
| **System: Ghost Access** | VLESS Reality :443 | Our production protocol |
| **System: Mesh Core** | Mesh routing without self-healing | Shows mesh overhead |
| **System: Mesh + MAPE-K** | Full self-healing mesh | Shows self-healing benefit |

## 🧬 2. Failure Taxonomy ($F1 - F10$)

All test suites and failure injectors MUST reference specific failure codes from this taxonomy:

| Failure ID | Category | Description |
| :--- | :--- | :--- |
| **$F1$** | `Node Crash` | Immediate process termination (`SIGKILL`) of an active routing/control node. |
| **$F2$** | `Kernel Panic` | Abrupt termination of eBPF XDP hook execution or driver queue reset. |
| **$F3$** | `Network Partition` | Complete split-brain subnet isolation via `iptables` / `tc`. |
| **$F4$** | `Synthetic Packet Loss` | Artificial packet drop ($10\% - 30\%$) via Linux `netem`. |
| **$F5$** | `High Latency / Jitter` | Injection of $100\text{ms} - 500\text{ms}$ artificial delay. |
| **$F6$** | `DNS Failure` | Upstream DNS resolution timeout or NXDOMAIN response injection. |
| **$F7$** | `Certificate Expiry` | SPIFFE/SPIRE SVID expiration during active mTLS session. |
| **$F8$** | `PQC Handshake Failure` | Corrupted ML-KEM-768 public key or mismatched signature verification. |
| **$F9$** | `Registry Loss` | Disconnection from node discovery registry. |
| **$F10$** | `Byzantine Node` | Malicious node broadcasting corrupted routing tables. |

## 🛠️ 3. Failure Injection Architecture (`inject_failure`)

The validation engine decoupling principle dictates that test suites **do not hardcode failure logic**. Instead, test scenarios invoke a standardized **Failure Injector**:

```
 ┌───────────────────────────┐
 │   Validation Test Suite   │
 └─────────────┬─────────────┘
               │ 1. Invokes inject_failure(F_x)
               ▼
 ┌───────────────────────────┐
 │ Failure Injector Engine   │ ---> Injects Netem / SIGKILL / SPIRE revoke
 └─────────────┬─────────────┘
               │ 2. Triggers Fault State
               ▼
 ┌───────────────────────────┐
 │ Telemetry Collector (Prom)│ ---> Records TTFB, RTT, Socket Survival
 └─────────────┬─────────────┘
               │ 3. Evaluates SLA Gates
               ▼
 ┌───────────────────────────┐
 │ PASS / WARN / FAIL Gate   │
 └───────────────────────────┘
```

## 🛡️ 4. Threats to Validity

Every validation report generated by this framework MUST analyze and report threats to validity:

### Internal Validity Threats
- **CPU Scheduling Jitter:** Variations in host Linux CPU scheduler (`CFS`) allocations.
- **Cache Effects:** L1/L2/L3 cache warming during repeated test runs.
- **eBPF Verifier Overhead:** Micro-delays during dynamic BPF bytecode loading.

### External Validity Threats
- **Kernel Dependency:** Results apply specifically to Linux Kernel $\ge 5.15$.
- **Un-tested Link Types:** Mobile 5G/4G, satellite (Starlink), or lossy wireless links require distinct test passes.

### Construct Validity Threats
- **Recovery Boundary Definition:** A state is defined as "Recovered" ONLY when TTFB returns within $p95$ baseline and zero socket drops are recorded for 10 consecutive seconds.

### Statistical Validity Threats
- Outliers MUST be analyzed via $IQR$ (Interquartile Range) filtering;
- Confidence intervals ($95\%$ CI) MUST be calculated via Bootstrap Resampling ($B = 1000$).

## 🚦 5. CI / SLA Pass-Fail Evaluation Gates

Automated pipelines MUST evaluate test outputs against strict SLA gates.

Every threshold MUST have a documented rationale in `validation/spec/sla_rationale.md`.

| Metric | PASS Criteria | WARNING Criteria | FAIL Criteria | Rationale Source |
| :--- | :--- | :--- | :--- | :--- |
| **Node Failure Recovery ($F1, F3$)** | $< 2.0 \text{ sec}$ | $2.0\text{s} - 5.0\text{s}$ | $\ge 5.0 \text{ sec}$ | Model hypothesis (see §5.1) |
| **Session Survival Rate ($F1, F4$)** | $\ge 99.5\%$ | $95.0\% - 99.4\%$ | $< 95.0\%$ | Production observation |
| **PQC Handshake Overhead ($F8$)** | $< 50 \text{ ms}$ | $50\text{ms} - 150\text{ms}$ | $> 150 \text{ ms}$ | Measured on target hardware |
| **Synthetic Loss Degradation ($F4$)** | $< 15\%$ TTFB drop | $15\% - 30\%$ drop | $> 30\%$ drop | TCP retransmission theory |

### 5.1 SLA Rationale (Model Hypotheses)

**IMPORTANT:** The following values are model-derived estimates, NOT measured facts.
They require experimental validation before being treated as SLA.

| Threshold | Derivation | Assumptions | Validation Required |
| :--- | :--- | :--- | :--- |
| Recovery < 2s | MAPE-K cycle (5s) + detection (~1s) + convergence (~0.5s) | Single node failure, stable network | Run Suite V1 N=30, measure actual T_recover |
| PQC < 50ms | ML-KEM-768 gen (~25ms) + encaps (~15ms) + verify (~5ms) | AMD Athlon 3000G, single thread | Benchmark on target hardware |
| Survival > 99.5% | 500 users × 0.5% = 2.5 affected | Single failure event | Run Suite V1 N=30, count survivals |

## 🏆 8. Architectural Validation Test Suites ($V1 - V7$)

1. **Suite V1 (F1/F3 Node Failure Recovery):** Measures exact $T_{detect}$ and $T_{reroute}$ upon node crash.
2. **Suite V2 (F4 Synthetic Loss Resilience):** Measures TTFB decay under $10\%, 20\%, 30\%$ loss.
3. **Suite V3 (F5/F6 Transport Fallback):** Measures Reality-443 fallback when UDP/443 is blocked.
4. **Suite V4 (Topology Reroute Convergence):** Measures route convergence under $N \ge 5$ topology shifts.
5. **Suite V5 (F8 PQC Key Exchange Overhead):** Measures ML-KEM-768 handshake cost vs ECDHE.
6. **Suite V6 (High Concurrency Scaling):** Measures $p99$ degradation from $10$ to $500$ connections.
7. **Suite V7 (F3 Network Partition & Reconciliation):** Measures split-brain reconciliation speed.
