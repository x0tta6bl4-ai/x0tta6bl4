# x0tta6bl4: Technical Trust & Security Whitepaper
**Version:** 3.3.1 (Reality-Aligned Draft)

Этот документ предназначен для CTO, CISO и архитекторов безопасности. Он описывает реализованные механизмы и текущие ограничения по внедрению.

## 1. Постквантовая криптография (PQC)
Control plane и data plane используют NIST-aligned алгоритмы:
*   **ML-KEM-768 (Kyber):** Обмен ключами, защита от сценария "harvest now, decrypt later".
*   **ML-DSA-65 (Dilithium):** Подпись управляющих команд и плейбуков.
*   **Статус:** Реализовано в кодовой базе через `liboqs`-совместимый слой; production требует доступности криптобэкенда в рантайме.

## 2. Zero-Trust Identity (SPIFFE/SPIRE)
Каждый узел (ноутбук, сервер, IoT-датчик) получает машинную идентичность:
*   **Workload ID:** Короткоживущие SVID-сертификаты с автообновлением по политике окружения.
*   **mTLS:** Взаимная аутентификация между сервисами в поддерживаемых контурах.
*   **Hardware Attestation:** Опциональная привязка к TPM/HSM.

## 3. Самовосстановление и Устойчивость (MAPE-K)
Система использует цикл Monitor -> Analyze -> Plan -> Execute:
*   **Monitor:** Телеметрия через eBPF/XDP там, где поддерживается средой выполнения.
*   **Analyze:** Детекция аномалий GraphSAGE-моделью и rule-based fallback.
*   **Plan:** Генерация ответных действий через PARL и автоматизацию плейбуков.
*   **Execute:** Перестройка маршрутов и восстановительные действия по политике self-healing.

## 4. Обход DPI и Маскировка (Anti-DPI)
Для ограничительных сетевых сред используются:
*   **Ghost Protocol (x0t-native):** Экспериментальный transport-профиль с паттерном пакетов, приближенным к WebRTC/SRTP.
*   **Differential Obfuscation:** Дополнительный слой вариативности размеров/интервалов пакетов.
*   **HTTP-Steganography:** Резервный канал маскировки для отдельных сценариев.

Эффективность Anti-DPI зависит от конкретного провайдера, региона и текущей политики фильтрации, поэтому требует полевой валидации в целевой сети.

## 5. Комплаенс и Аудит
*   **Immutable Audit Trail:** Модуль неизменяемого аудита (Merkle + IPFS-интеграция) реализован; для production необходима настройка внешних зависимостей и регламент хранения.
