# Моя персональная система фильтрации ИИ

Контекст проекта (зафиксировано на 90 дней):
- Роль: фаундер/CTO + hands-on инженер.
- Главная цель: выручка и MRR через коммерциализацию.
- Ближайший продуктовый контур: VPN/платежи/выдача конфига/лендинг.
- Ограничение: нельзя тратить фокус на все AI-релизы одновременно.

## 1) Операционная цель фильтра

Фильтр существует не для "быть в курсе", а для роста 3 метрик:
- MRR (месяц к месяцу).
- Конверсия в оплату (лендинг -> checkout -> paid).
- Время активации после оплаты (paid -> рабочий VLESS/QR).

Если AI-новость не влияет на эти 3 пункта, по умолчанию это шум.

## 2) Модель принятия решений: "важно / не важно"

Оценка релиза по 5 критериям (0/1/2 балла):
- Revenue Impact: влияет ли на MRR/конверсию/чек.
- Time-to-Value: даст ли эффект за 14 дней.
- Integration Cost: можно ли внедрить за <= 4 часа.
- Reliability/Compliance: снижает ли риски прод-цепочки.
- Reuse: можно ли применить в 2+ критичных процессах.

Порог:
- 8-10: внедрять сейчас.
- 5-7: в бэклог, пересмотр через неделю.
- 0-4: игнорировать.

Стоп-правило:
- Любой релиз с интеграцией > 1 дня или без измеримого KPI-эффекта автоматически отклоняется.

## 3) Еженедельный AI-агент-фильтр (под твой контекст)

Режим:
- 1 запуск в неделю, 45 минут лимит.
- Источники: официальные release notes инструментов, которые уже есть в стеке.
- Выход: 1 страница, 3 блока — "Важно", "Тест", "Игнор".

Промпт для агента:

```
Ты AI-фильтр для проекта x0tta6bl4.
Цель на 90 дней: рост MRR и ускорение paid->activation цепочки.
Текущие процессы: FastAPI API, Stripe webhook, Xray/VLESS provisioning, landing, Telegram bot, мониторинг.

Оцени каждый релиз по шкале 0..2:
Revenue Impact, Time-to-Value (<=14d), Integration Cost (<=4h), Reliability/Compliance, Reuse.

Верни только:
1) TOP-3 релиза к тесту (с ожидаемым KPI-эффектом),
2) 1 конкретный эксперимент на неделю,
3) список игнора (почему не влияет на цели).

Запрещено предлагать "изучить для общего развития" без KPI.
```

## 4) Метод тестирования релизов через реальные задачи

Тестируем не на демо, а на прод-процессе:
- Задача A: checkout -> webhook -> создание пользователя -> выдача VLESS.
- Задача B: лендинг -> checkout-конверсия.
- Задача C: сокращение ручной поддержки по подключению.

Дизайн теста (каждый релиз):
- Гипотеза: "если внедрить X, то метрика Y улучшится на Z%".
- Лимит времени: 30-90 минут.
- A/B или before/after на 1 реальном флоу.
- Решение: Keep / Kill в ту же неделю.

## 5) Как отличать benchmark-релизы от business-релизов

Benchmark-релиз:
- Акцент на MMLU/arena/latency без workflow-кейсов.
- Нет стабильного SDK/миграционного пути/цены.
- Нет эффекта на текущую воронку выручки.

Business-релиз:
- Улучшает time-to-market или revenue-метрику в текущем флоу.
- Имеет стабильный API/SDK и понятную эксплуатацию.
- Даёт измеримый эффект за 1-2 недели.

## 6) Рабочий регламент против FOMO

- По умолчанию "нет" всему новому.
- Максимум 1 AI-эксперимент в неделю.
- Никаких "переездов стеков" без подтвержденного прироста ключевой метрики.
- Любая новая AI-идея должна иметь owner, срок и KPI.

## 7) Что делать с найденным релизом (решение за 5 минут)

- Шаг 1: поставить баллы по 5 критериям.
- Шаг 2: если < 8 — сразу в ignore/backlog.
- Шаг 3: если >= 8 — сформировать 1-недельный micro-test.
- Шаг 4: в конце недели принять Keep/Kill, без "вечно тестируем".

## 8) KPI самой системы фильтра

Через 4 недели:
- <= 2 часов/неделя на AI-новости суммарно.
- >= 3 убранных шумовых направления.
- >= 1 внедрение с измеримым эффектом на ключевую метрику.
- Снижение тревожного потребления новостей (самооценка 1-10) минимум на 2 пункта.
