# NL VPN incident, 2026-05-26

## Короткий вывод

VPN перестал работать не из-за локальной настройки Xray/x-ui. VPS был выключен со стороны гипервизора провайдера.

Главная строка в журнале:

```text
May 26 10:18:29 01164.com qemu-ga[785]: info: guest-shutdown called, mode: powerdown
May 26 10:19:14 01164.com systemd-logind[641]: System is powering down (hypervisor initiated shutdown).
```

`hypervisor initiated shutdown` означает, что выключение пришло с хост-машины провайдера, а не изнутри VPS.

## Время

- Предыдущая загрузка: `2026-05-25 18:43:23 UTC` - `2026-05-26 10:21:04 UTC`
- Текущая загрузка: с `2026-05-27 02:50:41 UTC`
- Сервер был недоступен примерно с `2026-05-26 10:21 UTC` до `2026-05-27 02:50 UTC`
- В локальном времени UTC+3: примерно с `2026-05-26 13:21` до `2026-05-27 05:50`

## Что происходило перед выключением

`sar` показывает сильную деградацию хоста провайдера:

```text
09:30 UTC  %steal=11.06
09:40 UTC  %steal=23.86
09:50 UTC  %steal=27.51
10:00 UTC  %steal=38.82
10:10 UTC  %steal=43.78
```

`%steal` - это время, когда VPS хотел использовать CPU, но гипервизор его не дал. Для 2 CPU это очень плохо.

Также выросла очередь и задержка диска:

```text
10:10 UTC  ldavg-1=31.09, runq-sz=36
10:10 UTC  vda await=187.19 ms
```

Вывод: перед выключением была перегрузка/деградация на стороне хост-машины или storage у провайдера.

## Локальные усилители проблемы

Найдены два cron-скрипта, которые могли ухудшать ситуацию во время деградации:

- `/usr/local/bin/proactive_auditor.py`
- `/usr/local/bin/auto-port-fallback.sh`

Они могли часто делать `systemctl restart x-ui` / `systemctl restart warp-svc` без паузы и без проверки нагрузки.

Это не первопричина выключения VPS, но это могло ускорять деградацию сервиса во время проблем у провайдера.

## Что изменено 2026-05-27

В оба скрипта добавлена защита:

- cooldown 30 минут между restart;
- пропуск restart при высокой нагрузке VPS;
- lock для `auto-port-fallback.sh`, чтобы не было параллельных запусков;
- более явные записи в `/var/log/x0tta6bl4_audit.log`;
- в Python убран `shell=True` для опасных команд и добавлены таймауты.

Бэкапы на сервере:

```text
/usr/local/bin/proactive_auditor.py.bak-20260527-incident
/usr/local/bin/auto-port-fallback.sh.bak-20260527-incident
```

## Отдельная не первичная проблема, исправлено

На сервере есть конфликт сетевой настройки:

- `systemd-networkd` управляет `eth0` через netplan DHCP;
- `networking.service` / ifupdown тоже пытается поднять `eth0`.

Из-за этого были failed units:

```text
ifup@eth0.service
networking.service
```

Это не причина выключения VPS. Исправлено 2026-05-27:

```text
cp -a /etc/network/interfaces /etc/network/interfaces.bak-20260527-network-conflict
systemctl disable networking.service
systemctl reset-failed networking.service ifup@eth0.service
```

Текущая сеть после изменения осталась активной:

```text
default via 89.125.1.1 dev eth0 proto dhcp src 89.125.1.107 metric 100
89.125.1.107/24 on eth0
```

`systemctl --failed` после исправления: `0 loaded units listed`.
